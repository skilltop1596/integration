**一体化业务管控平台需求文档（.NET+Vue3）**

**1. 文档概述**

**1.1 文档目的**

本文档为一体化业务管控平台的完整需求文档，明确系统的功能范围、业务逻辑、技术架构、数据库设计及开发规范，作为.NET后端开发、Vue3前端开发、测试及部署的唯一依据，确保开发过程贴合业务需求，实现“客户CRM管理+工单/业务反馈+项目管理+功能模块结构化文档归档”的全业务闭环，且不依赖外部项目管理工具（如禅道），保证系统独立可运行。

**1.2 项目背景**

现有业务中，客户管理、工单反馈、项目推进、需求文档归档存在分散管理、信息脱节、查阅不便等问题，需搭建一套独立的一体化业务管控平台，整合CRM客户管理、工单反馈处理、项目全流程管理、功能模块结构化文档归档四大核心能力，采用.NET+Vue3技术栈，实现业务流程标准化、数据集中化、文档结构化，提升业务效率，降低管理成本。

**1.3 技术栈要求（强制）**

• **后端技术栈**：.NET 6（推荐）/ .NET 8，采用ASP.NET Core Web API架构，ORM框架使用SqlSugar（高效、易扩展，适配多数据库），依赖注入、AOP面向切面编程实现日志、权限拦截，集成Swagger生成API文档，可选集成SignalR实现实时消息推送。

• **前端技术栈**：Vue3（Composition API + \<script setup>语法糖），UI组件库使用Element Plus，构建工具使用Vite5，状态管理采用Pinia，路由管理使用Vue Router，HTTP请求使用Axios，文档在线预览集成vue-pdf、xlsx等插件，遵循Anthony Fu Vue3开发规范保证代码可维护性。

• **数据库**：SQL Server 2019（强制），支持数据备份、事务管理，适配SqlSugar ORM框架，确保数据一致性和安全性，遵循SQL Server 2019数据库设计规范，表结构设计符合3NF范式要求。

• **部署环境**：后端部署在Windows Server（IIS），前端部署在Nginx，数据库独立部署，支持Docker容器化部署（可选），适配SQL Server 2019运行环境。

**预期目标**

• 实现客户、工单、项目、文档的集中管理，解决信息脱节问题，提升数据完整性，降低数据重复率；

• 实现需求文档按功能模块结构化归档，支持版本管理、全文检索，提升查阅效率；

• 实现业务流程闭环，从客户建档、工单提交到项目推进、文档归档全流程可追溯；

• 系统操作简洁，适配不同角色（销售、客服、产品、管理员）的使用场景，降低学习成本；

• 系统稳定、高效，支持多用户并发操作，数据安全可追溯，核心接口响应及时；

• 接口设计遵循RESTful规范，便于后续扩展和维护，支持前端与后端高效通信；

• 数据库设计贴合SQL Server 2019特性，表结构规范，支持数据备份与恢复，保障数据安全。

**2. 核心业务需求**

系统核心分为五大模块：客户CRM管理模块、工单&业务反馈模块、项目管理模块、功能模块&文档中心模块、系统后台管理模块，各模块独立运行且相互关联，实现业务闭环。

**2.1 客户CRM管理模块**

**2.1.1 功能需求**

1. **客户档案管理**：支持客户新增、编辑、删除、查看，录入客户基本信息（客户编号、名称、行业、等级、来源、联系方式、地址等），客户编号自动生成（规则：客户类型缩写+年月日+序号），支持客户信息导入导出（Excel格式）；录入时需进行字段校验，核心字段（客户名称、联系方式）强制录入，格式符合标准化要求，如手机号需匹配正则表达式校验规则，邮箱需符合规范格式。

2. **联系人管理**：每个客户可关联多个联系人，录入联系人姓名、职位、手机号、邮箱、备注等信息，支持联系人新增、编辑、删除，可快速关联客户跟进记录；联系人信息需与客户信息关联校验，确保数据一致性。

3. **客户跟进记录**：业务/客服人员可添加客户跟进纪要，记录跟进时间、跟进方式（电话、微信、拜访等）、跟进内容、下次跟进计划，跟进记录关联客户和对应项目/功能模块，支持按客户、时间范围检索；跟进方式采用枚举值控制，禁止手动输入，确保数据标准化。

4. **客户标签&分层**：支持自定义客户标签（如潜在客户、合作客户、高价值客户等），支持客户分层管理，可按标签、分层筛选客户，便于精准跟进；可通过多维度组合匹配实现客户模糊去重，相似度≥85%视为重复并提示人工复核。

5. **客户权限隔离**：采用数据权限控制，业务员只能查看/操作自己负责的客户，管理员可查看所有客户，防止客户数据泄露；管理层查看客户详情时，强制显示历史跟进记录字段，普通销售仅可见自身记录。

**2.2.2 业务约束**

• 客户名称、客户编号不可重复，客户编号自动生成，不可手动修改；客户编号命名遵循规范，采用富有意义的英文缩写+日期+序号组合，避免使用保留字；

• 客户删除需先解绑关联的项目、工单、文档，否则无法删除；删除操作仅设置删除标志，不做物理删除，保留历史数据便于追溯；

• 跟进记录提交后不可删除，仅可编辑补充，确保跟进痕迹可追溯；跟进记录需包含建档人员、建档时间、最后修改人员、最后修改时间等基础字段；

• 客户信息导入时需进行数据校验，不符合规则的数据（如手机号格式错误、必填字段缺失）提示错误，不予导入；导入数据需与系统现有数据进行去重校验，避免重复录入；

• 若客户行业选择“制造业”，则强制要求填写生产规模和主要产品线；当客户进入报价阶段时，强制要求填写预算范围和决策周期。

**2.2 工单&业务反馈模块**

**2.2.1 功能需求**

1. **反馈提交入口**：支持多来源反馈提交（客户自主反馈、客服收集、业务销售提报、内部员工建议、售后报修），提交时需选择反馈类型、优先级、所属项目、所属功能模块，填写反馈内容、附件（可选）；提交时进行必填字段校验，确保核心信息完整。

2. **反馈分类与分级**：反馈类型分为Bug问题、功能建议、流程优化、操作咨询、数据问题；优先级分为紧急、高、中、低，提交时可手动选择，紧急反馈自动触发消息通知；反馈类型和优先级采用枚举值控制，统一选项规范，避免手动输入导致的数据混乱。

3. **反馈状态流转**：支持状态闭环流转，流转路径：提交→受理（产品/客服）→需求梳理（产品）→纳入规划（产品）→已落地（产品确认）→已办结（反馈人确认）；支持手动修改状态，每步状态变更需填写备注，状态变更记录自动留存，便于追溯。

4. **反馈关联绑定**：每条反馈必须绑定所属项目和功能模块，可关联客户、联系人，支持关联已有文档，便于追溯反馈来源和处理依据；关联数据需与对应表数据校验，确保关联关系合法有效。

5. **重复反馈合并**：支持手动合并相似反馈（同一功能模块、同一问题），合并后以主反馈为准，子反馈关联主反馈，避免文档和反馈冗余；合并操作需记录操作人、操作时间和合并原因，便于后续核查。

6. **反馈查询与统计**：支持按反馈类型、优先级、状态、项目、功能模块、时间范围检索反馈，支持统计反馈处理时效、各类型反馈数量，生成简易报表；报表数据可导出Excel格式，便于数据统计分析。

**2.2.2 业务约束**

• 反馈提交时，所属项目、功能模块、反馈类型为必填项，紧急反馈需填写紧急原因；反馈内容长度不低于20字，确保反馈信息完整；

• 反馈状态流转不可逆（如已办结状态不可回退到受理状态），特殊情况需管理员手动干预；状态变更需记录操作日志，包含操作人、操作时间、旧状态、新状态和操作备注；

• 反馈附件支持Word、PDF、图片、Excel格式，单个文件大小不超过100MB，最多可上传5个附件；附件上传后自动记录上传人、上传时间，支持在线预览和下载；

• 反馈处理时效有明确要求：紧急反馈2小时内受理，高优先级反馈4小时内受理，中低优先级反馈24小时内受理，超时未受理自动触发提醒通知；

• 已办结的反馈不可修改，仅可查看；若需重新处理，需新建反馈并关联原反馈记录，确保反馈处理轨迹可追溯。

**2.3 项目管理模块**

**2.3.1 功能需求**

1. **项目基础管理**：支持项目新增、编辑、删除、查看，录入项目基本信息（项目名称、项目编号、项目类型、开始时间、结束时间、负责人、关联客户、项目描述等），项目编号自动生成（规则：项目类型缩写+年月日+序号）；项目编号命名遵循规范，避免使用保留字，确保唯一性。

2. **项目分层分类**：项目类型分为定制项目、迭代版本、售后运维、需求优化；项目状态分为筹备中、需求梳理、开发中、验收中、已结项，支持按类型、状态筛选项目；项目类型和状态采用枚举值控制，统一选项规范，便于数据统计。

3. **项目角色权限**：为每个项目配置角色成员（项目经理、产品、业务对接人、客服对接人、只读查看人），不同角色拥有不同操作权限（如项目经理可编辑项目信息、维护里程碑，只读查看人仅可查看项目详情）；角色权限遵循RBAC模型，权限分配需记录操作日志。

4. **项目里程碑管理**：支持自定义项目里程碑（如需求定稿、原型完成、开发启动、测试完成、客户验收、正式上线），设置每个里程碑的计划时间、实际完成时间、负责人、备注，里程碑延期自动触发消息通知（通知项目经理和相关成员）；里程碑计划时间不可晚于项目结束时间，实际完成时间需如实填写，延期需说明原因。

5. **项目聚合视图**：每个项目页面聚合显示关联的客户、工单反馈、功能模块、文档，一屏查看项目全信息，支持快速跳转至对应模块；聚合数据实时同步，确保信息一致性。

6. **项目归档与封存**：项目结项后自动归档，归档后不可随意编辑项目信息和关联内容，仅可查看，支持按归档时间检索历史项目；归档项目需完善验收资料，确保项目闭环；项目归档后数据不做物理删除，仅设置归档标志，保留历史数据。

**2.3.2 业务约束**

• 项目名称、项目编号不可重复，项目编号自动生成，不可手动修改；项目信息需包含建档人员、建档时间、最后修改人员、最后修改时间等基础字段；

• 项目删除需先结项，结项后不可删除，仅可归档；删除操作需管理员确认，避免误操作；

• 里程碑计划时间不可晚于项目结束时间，实际完成时间不可早于计划时间（特殊情况需填写备注）；里程碑状态变更需由负责人确认，确保信息准确；

• 项目负责人变更需记录变更原因和变更时间，关联项目历史记录，便于追溯；

• 未结项的项目不可归档，归档前需完成所有关联工单、文档的整理和确认，确保项目数据完整。

**2.4 功能模块&文档中心模块（核心）**

**2.4.1 功能模块库管理**

1. **全局标准模块字典**：建立全局统一的功能模块字典，支持一级模块、二级子模块分级（如一级模块：客户管理；二级子模块：客户新增、客户编辑、客户公海），管理员可新增、编辑、停用模块，模块信息包括模块名称、模块编码、父模块ID、负责人、模块描述、模块状态；模块编码命名规范，采用英文缩写+序号组合，确保唯一性，避免使用保留字。

2. **模块复用与复制**：同类型项目可直接复制整套功能模块结构（一级+二级），无需从零搭建，复制后可根据项目需求修改模块细节；复制操作需记录操作人、操作时间，便于追溯。

3. **模块状态管理**：模块状态分为未规划、需求编写中、已定稿、迭代中、已上线、已下线，支持按状态筛选模块，模块状态变更需由负责人确认；状态变更需记录操作日志，包含操作人、操作时间、旧状态、新状态和变更说明。

4. **项目模块关联**：每个项目可引用全局模块字典中的模块，关联后模块自动归属该项目，支持为项目自定义新增专属模块（仅该项目可用）；专属模块需标注项目归属，避免与全局模块混淆。

**2.4.2 文档中心管理**

1. **文档目录结构化**：按项目+功能模块自动生成标准化文档目录，目录结构固定为：项目总档→基础资料→业务架构→功能模块分类→需求变更→验收交付→培训手册，每个功能模块下自动生成固定子文件夹（模块总体需求、各子功能需求、原型、流程图等）；目录结构遵循标准化规范，便于文档统一管理和查阅。

2. **文档上传与归类**：上传文档时，强制选择所属项目、所属功能模块、文档类型（需求文档、原型附件、流程图、变更记录、验收资料等），支持批量上传，自动记录上传人、上传时间、版本号；文档命名规范，采用“文档类型+模块名称+版本号”格式，避免重复。

3. **文档版本管理**：同一份文档多次上传自动保留版本（版本号按1.0、1.1、2.0递增），支持查看历史版本、版本对比（高亮差异）、回滚至历史版本，版本变更需填写变更说明；版本记录需包含版本号、变更人、变更时间、变更说明，便于追溯版本迭代过程。

4. **文档在线预览**：支持Word、PDF、Excel、图片、流程图等格式文档在线预览，无需下载，提升查阅效率；支持文档批注（可选），批注内容关联版本，不可修改；批注需记录批注人、批注时间，便于协作沟通。

5. **文档检索与筛选**：支持按项目、功能模块、文档名称、文档类型、上传人、时间范围全文检索，支持多条件组合筛选，检索结果按相关性排序；检索响应时间≤2秒，确保检索效率。

6. **文档操作留痕**：所有文档的上传、编辑、删除、下载、查看、版本变更操作均记录日志（操作人、操作时间、操作内容），可追溯操作记录；日志保存周期不少于1年，便于数据安全审计。

7. **需求变更台账**：单独建立需求变更台账，按功能模块归档，记录变更内容、变更原因、变更申请人、客户确认情况、变更影响范围、变更时间，支持关联对应文档和反馈；需求变更需经产品负责人确认后才可归档，变更内容需关联客户确认记录（可选），确保变更合规。

**2.4.3 业务约束**

• 全局模块字典中，模块名称、模块编码不可重复，父模块ID不可指向自身；模块编码命名遵循规范，避免使用保留字和特殊字符；

• 文档上传时，所属项目、功能模块、文档类型为必填项，文档名称不可重复（同一模块下）；文档大小不超过200MB，支持常用格式，特殊格式需提前报备；

• 已归档项目的文档不可编辑、删除，仅可查看和下载；文档删除需管理员确认，删除后仅设置删除标志，不做物理删除，保留历史版本；

• 需求变更台账需经产品负责人确认后才可归档，变更内容需关联客户确认记录（可选）；变更记录需包含建档人员、建档时间、最后修改人员、最后修改时间等基础字段；

• 文档版本回滚仅可回滚至历史已保存版本，回滚操作需填写原因，记录操作人、操作时间，回滚后原版本不删除，仍可查看。

**2.5 系统后台管理模块**

**2.5.1 功能需求**

1. **组织架构管理**：支持部门新增、编辑、删除、排序，录入部门名称、父部门、负责人、部门描述，支持查看部门下所有员工；部门编码命名规范，采用英文缩写+序号组合，确保唯一性，避免使用保留字；部门删除需先解绑关联员工，否则无法删除。

2. **业务字典配置**：支持自定义业务字典（如项目类型、反馈类型、模块状态、工单状态、客户等级、客户来源等），可新增、编辑、停用字典项，字典项支持排序，全局统一使用；字典项编码遵循规范，确保唯一性，便于数据关联和检索。

3. **日志中心**：包含操作日志、登录日志、数据变更日志，操作日志记录所有用户的功能操作，登录日志记录用户登录/退出时间、登录IP，数据变更日志记录客户、项目、文档等核心数据的变更记录，支持按时间、用户、操作类型检索；日志保存周期不少于1年，不可删除，便于数据安全审计和问题排查。

4. **消息通知管理**：支持全局消息通知配置，可配置通知场景（里程碑延期、紧急反馈、文档更新、工单状态变更等），通知方式为系统内通知，支持查看未读/已读通知、标记已读、删除通知，支持批量操作；通知记录保留3个月，过期自动归档。

5. **数据导入导出**：支持客户、项目、工单、文档目录等核心数据的导入导出（Excel格式），导入时支持数据校验，不符合规则的数据提示错误，导出时支持筛选条件导出；导入导出操作需记录操作日志，包含操作人、操作时间、操作内容和数据量。

6. **系统配置**：支持配置系统名称、Logo、登录页信息、文件上传路径、数据备份周期、密码加密规则等，配置变更需管理员确认；配置信息需备份，变更后可回滚至历史配置；数据备份支持自动备份和手动备份，自动备份周期可配置，备份文件保存周期不少于3个月，加密保存。

7. **数据权限管理**：支持按部门、角色分配数据权限，控制用户可查看、操作的数据范围（如业务员仅可见自身负责的客户、项目）；权限分配需记录操作日志，权限变更即时生效；支持权限预览，便于管理员核查权限分配情况。

**2.5.2 业务约束**

• 系统管理员账号不可禁用、不可删除，默认拥有所有权限；管理员操作需进行二次确认，避免误操作；

• 业务字典项删除需先解绑关联的数据（如项目类型字典项删除需先确保无关联项目），否则无法删除；字典项停用后，不可新增关联数据，但历史关联数据保留；

• 日志记录不可删除，仅可查看，保存周期不少于1年；日志数据需加密存储，防止篡改，确保审计有效性；

• 数据备份自动执行，备份文件保存周期不少于3个月，支持手动备份；备份文件需加密保存，防止数据泄露，备份恢复需管理员确认；

• 系统配置变更后需重启系统方可生效，变更前需备份当前配置，便于异常时回滚；文件上传路径变更后，历史文件需手动迁移，确保文件可正常访问。

**3. 非功能需求**

**3.1 性能需求**

• 并发性能：支持至少50人同时在线操作，核心接口（客户查询、文档上传/预览、工单提交）响应时间≤1秒，非核心接口响应时间≤3秒；适配SQL Server 2019的并发处理能力，避免并发冲突。

• 存储性能：支持至少1000个客户、500个项目、10000份文档、5000条工单反馈的存储，文档存储支持扩容；数据库表结构设计合理，建立合适索引，提升查询性能，索引命名遵循规范（IX\_表名\_字段名）。

• 稳定性：系统连续运行无故障时间≥72小时，故障恢复时间≤30分钟，支持异常日志记录，便于排查问题；数据库支持事务管理，确保数据一致性，避免数据丢失或错乱。

• 检索性能：全文检索响应时间≤2秒，支持模糊检索、精确检索，检索结果准确无误；采用合理的检索索引设计，适配SQL Server 2019的检索特性。

**3.2 安全需求**

• 权限安全：严格的权限控制，按角色分配功能权限和数据权限，防止越权操作，敏感操作（如删除客户、修改权限）需二次确认；权限校验贯穿所有操作流程，确保每一步操作都有对应权限。

• 数据安全：核心数据（客户、合同、文档）支持加密存储，定期自动备份，备份文件加密保存，防止数据泄露、丢失；数据库采用SQL Server 2019的加密功能，敏感字段加密存储，日志记录所有数据操作，便于追溯。

• 接口安全：所有API接口支持Token认证，防止非法调用，接口请求频率限制（每分钟≤60次），防止恶意攻击；接口参数需进行校验，避免SQL注入等安全隐患。

• 操作安全：所有敏感操作（删除、修改核心数据）记录日志，可追溯，支持操作撤销（部分操作）；操作日志需包含操作人、操作时间、操作IP、操作内容，确保操作可审计。

**3.3 易用性需求**

• 界面设计：简洁、直观，符合主流系统操作习惯，菜单层级不超过3级，核心功能一键可达；界面风格统一，按钮、表单、表格等组件样式一致，提升用户体验。

• 操作便捷：支持批量操作（批量上传文档、批量导出数据、批量处理工单），支持快捷键操作，减少重复操作；常用功能可添加到收藏夹，快速访问。

• 提示友好：操作失败、数据校验错误时，给出清晰的提示信息，引导用户正确操作；提示信息简洁明了，避免专业术语过多，便于所有角色理解。

• 适配性：支持电脑端访问（Chrome、Edge、Firefox等主流浏览器），适配不同屏幕分辨率（1920×1080、1366×768等）；文档预览适配不同格式，确保预览效果清晰。

**3.4 可扩展性需求**

• 架构可扩展：采用模块化、分层架构，后端支持插件式开发，前端支持组件复用，便于后续新增功能模块（如合同管理、回款管理）；模块之间低耦合，新增模块不影响现有功能。

• 接口可扩展：API接口设计遵循RESTful规范，预留接口扩展字段，便于后续对接外部系统（如后续对接禅道）；接口版本管理清晰，升级接口不影响旧版本接口使用。

**4. 技术架构设计**

**4.1 整体架构（分层架构）**

**4.1.1 后端架构（.NET）**

1. **表现层（API层）**：基于ASP.NET Core Web API，负责接收前端请求、返回响应结果，集成Swagger生成API文档，处理Token认证、请求参数校验、异常统一处理，接口设计遵循RESTful规范，支持跨域请求处理，可选集成SignalR实现实时消息推送；适配SQL Server 2019的数据库连接方式，确保接口与数据库高效通信。

2. **业务逻辑层（BLL层）**：负责核心业务逻辑处理，如客户管理、工单流转、项目管理、文档归档等，实现业务规则校验、数据处理，调用数据访问层接口，采用依赖注入实现解耦，集成AOP实现日志、事务、权限拦截；事务管理适配SQL Server 2019的事务特性，确保业务数据一致性。

3. **数据访问层（DAL层）**：基于SqlSugar ORM框架，负责与SQL Server 2019数据库交互，实现数据的CRUD操作，封装通用数据访问接口，适配SQL Server 2019的语法特性和数据类型，处理数据分页、筛选、排序等操作，确保数据访问高效、安全；索引设计遵循SQL Server 2019规范，提升查询性能。

4. **实体层（Model层）**：定义核心实体类（如客户、项目、工单、文档等），与SQL Server 2019数据库表一一对应，包含实体属性、数据校验规则，同时定义DTO类（数据传输对象），用于前后端数据交互，避免实体类直接暴露；实体类字段类型适配SQL Server 2019的数据类型，确保数据映射准确。

5. **公共层（Common层）**：封装通用工具类（如加密工具、日志工具、Excel导入导出工具、文件处理工具）、常量定义、异常处理、全局配置，供其他层调用，统一通用功能实现，提升代码复用性；包含SQL Server 2019数据库连接工具，确保数据库连接稳定。

**4.1.2 前端架构（Vue3）**

1. **视图层（View层）**：基于Vue3 + Element Plus，负责页面渲染，包含页面组件、布局组件、通用组件（如分页、弹窗、表单），采用\<script setup>语法糖编写，遵循Anthony Fu Vue3开发规范，确保代码简洁可维护；表单组件支持数据校验，与后端校验规则保持一致。

2. **路由层（Router层）**：基于Vue Router，负责页面路由管理，实现路由跳转、路由守卫（权限控制）、路由懒加载，按模块划分路由，支持嵌套路由，确保路由结构清晰；路由权限与后端权限同步，未授权路由无法访问。

3. **状态管理层（Store层）**：基于Pinia，负责全局状态管理，如用户信息、权限信息、通知消息等，实现状态的存储、修改、监听，避免组件间状态传递繁琐，提升状态管理效率；状态数据同步后端数据，确保数据一致性。

4. **请求层（API层）**：基于Axios，封装HTTP请求，实现请求拦截（添加Token、请求参数处理）、响应拦截（异常处理、数据格式化），按模块划分API接口，统一请求路径和响应格式，适配后端RESTful API；请求失败时给出友好提示，支持重试机制。

5. **工具层（Utils层）**：封装通用工具函数（如日期格式化、数据校验、文件处理、权限判断），供组件和API层调用，统一工具实现，提升代码复用性；包含文档预览工具，适配不同格式文档预览需求。

6. **公共组件层（Components层）**：封装通用组件（如分页组件、表单组件、弹窗组件、文档预览组件），支持组件复用，统一组件样式和交互，提升开发效率和页面一致性；组件可灵活配置，适配不同业务场景。

**5.3 数据库约束与规范**

• **命名规范**：数据库名称、表名、字段名均采用英文大写，表名前缀按模块划分（TB\_SYS\_系统管理、TB\_CRM\_客户管理、TB\_WORKORDER\_工单反馈、TB\_PROJECT\_项目管理、TB\_MODULE\_功能模块、TB\_DOC\_文档管理），字段名遵循“驼峰命名”，避免使用SQL Server 2019保留字，确保命名规范易懂。

• **字段约束**：所有表主键均为ID（bigint，自增），核心字段（如名称、编号）设置非空、唯一约束；外键关联设置级联查询，禁止级联删除，避免数据误删；日期字段默认值为当前时间，状态字段采用枚举值控制，确保数据标准化。

• **索引规范**：核心查询字段（如客户编号、项目编号、文档名称）建立普通索引，外键字段建立索引，索引命名遵循“IX\_表名\_字段名”格式；避免过度索引，确保数据库查询性能，适配SQL Server 2019索引优化特性。

• **数据安全**：所有表均包含IsDelete字段（bit类型），删除操作仅设置删除标志，不做物理删除；核心敏感字段（如联系方式、文档路径）支持加密存储，定期自动备份数据库，备份文件加密保存，保障数据安全。

• **范式要求**：表结构设计遵循3NF范式，避免数据冗余；合理设计关联表，减少多表联查复杂度，提升数据库操作效率，适配SQL Server 2019的查询优化机制。
